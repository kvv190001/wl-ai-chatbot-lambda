What is Commodity Price Modeling?

Commodity Price Modeling is an AI-powered solution that uses machine learning models to predict commodity prices for procurement decisions. Our system transitions from costly notebook-based approaches to an efficient API-based architecture, enabling real-time price predictions at scale. This helps organizations make informed purchasing decisions by running ad-hoc what-if scenarios to predict prices based on various market conditions and part specifications.

What problem does the Commodity Price Modeling system solve?

Traditional notebook-based approaches on platforms like Databricks are costly and inefficient, especially when scaling to approximately 200 concurrent requests. Each execution requires spinning up clusters, leading to high operational costs and slow response times. Our API-based architecture eliminates these bottlenecks by providing a scalable, cost-effective solution that delivers real-time price predictions without the overhead of cluster management.

How does the API-based architecture improve performance?

The API-based architecture eliminates the need to spin up Databricks clusters for each prediction request. Instead, the system uses multiple endpoints to receive input requests, fetch lookup data, invoke ML models through Databricks Model Serving, and return price predictions to client applications. This approach dramatically improves scalability, reduces operational costs, and enables real-time performance for concurrent users.

What is the purpose of the Ideation Accelerator project?

The Ideation Accelerator project exposes current price prediction ML models to applications via APIs, enabling ad-hoc what-if scenario analysis for price prediction. This initiative transforms how organizations interact with ML models, moving from manual notebook execution to automated, scalable API calls that support business decision-making in real-time.

How many concurrent requests can the system handle?

The new API-based architecture is designed to efficiently handle approximately 200 concurrent requests, a significant improvement over the notebook-based approach. By eliminating cluster spin-up overhead and leveraging Databricks Model Serving endpoints, the system can scale horizontally to meet demand while maintaining consistent performance and cost efficiency.

What technology stack powers the Commodity Price Modeling system?

The backend uses Python 3.x with Flask 3 (including flask-cors, requests, and python-dotenv) and Jinja2 for templating. The frontend leverages HTML, CSS, and vanilla JavaScript with Bootstrap 5 and Font Awesome via CDN. Data access is handled through databricks-sql-connector for SKU lookups and Databricks Model Serving with 12 endpoints. The system includes in-memory caching for SKU data with TTL, configuration via .env and app.yaml files, and optional prototype tools like FastAPI with Uvicorn, httpx, asyncio job queues, Server-Sent Events (SSE), and Databricks Jobs API for offloading.

What is Databricks Model Serving?

Databricks Model Serving is a managed service that hosts machine learning models as REST API endpoints. In our Commodity Price Modeling system, we use 12 Model Serving endpoints to expose trained price prediction models. This eliminates the need to run notebooks for each prediction, providing low-latency, scalable access to ML models through simple API calls.

How does the system fetch lookup data?

The system uses databricks-sql-connector to query Databricks SQL Warehouse for SKU lookup data. This data includes part specifications, historical pricing information, and other attributes needed for accurate price predictions. An in-memory cache with Time-To-Live (TTL) reduces database queries and improves response times for frequently accessed SKU data.

What is the role of in-memory caching in the system?

In-memory caching stores frequently accessed SKU lookup data with a Time-To-Live (TTL) mechanism. This reduces the number of queries to the Databricks SQL Warehouse, decreases response latency, and improves overall system performance. The cache automatically expires and refreshes data based on the configured TTL, ensuring predictions use reasonably current information while maintaining speed.

How does the Flask backend work?

The Flask 3 backend serves as the core application server, handling HTTP requests from client applications. It uses flask-cors for cross-origin resource sharing, enabling frontend applications to call the API from different domains. The requests library facilitates communication with Databricks Model Serving endpoints, while python-dotenv loads configuration from .env files. Jinja2 templates provide optional server-side rendering capabilities.

What are the multiple endpoints in the system architecture?

The system architecture includes multiple endpoints for different functions: input endpoints receive price prediction requests from client applications, lookup endpoints fetch SKU and part data from Databricks SQL Warehouse, model serving endpoints invoke the 12 Databricks ML models for price predictions, and response endpoints return formatted prediction results to requesting applications. This separation of concerns enables scalability and maintainability.

How does the system process a price prediction request?

When a client submits a price prediction request, the system receives the input through an API endpoint, parses the request to extract part specifications and scenario parameters, fetches relevant lookup data from the cache or Databricks SQL Warehouse, invokes the appropriate ML model through Databricks Model Serving with the enriched data, receives the predicted price from the model, and returns the formatted prediction result to the client application. This entire process happens in real-time without cluster spin-up delays.

What is the difference between the notebook-based and API-based approaches?

The notebook-based approach requires manually running Databricks notebooks for each prediction, spinning up compute clusters for execution, waiting for cluster initialization and notebook execution, and manually extracting results. This is costly and slow at scale. The API-based approach provides instant access through REST endpoints, no cluster management overhead, automatic scaling to handle concurrent requests, and programmatic integration with business applications. The API approach reduces costs and enables real-time decision-making.

How many ML models are used for price prediction?

The system uses 12 machine learning models exposed through Databricks Model Serving endpoints. These models may represent different commodity categories, prediction methodologies, or market segments. By leveraging multiple models, the system can provide specialized predictions based on part characteristics, market conditions, and other factors that influence commodity pricing.

What is the role of Databricks SQL Warehouse?

Databricks SQL Warehouse provides the data foundation for price predictions by storing SKU information, part specifications, historical pricing data, and other attributes needed for model inference. The system queries the SQL Warehouse to enrich prediction requests with relevant context before invoking ML models. This optional component can be replaced with other data sources depending on organizational needs.

How does configuration management work?

Configuration is managed through .env files for environment-specific settings like database connections, API keys, and model endpoint URLs, and app.yaml files for application-level configuration including caching parameters, timeout settings, and feature flags. This separation allows easy deployment across development, staging, and production environments without code changes.

What are the optional prototype tools mentioned?

For advanced use cases and prototyping, the system can leverage FastAPI with Uvicorn for high-performance async API serving, httpx for async HTTP requests to model endpoints, asyncio job queues for handling burst traffic and request prioritization, Server-Sent Events (SSE) for real-time prediction status updates to clients, and Databricks Jobs API for offloading long-running or batch prediction workloads. These tools extend the core Flask implementation for specific performance or functionality requirements.

What is FastAPI and when would you use it?

FastAPI is a modern, high-performance Python web framework for building APIs with automatic OpenAPI documentation and async support. In the Commodity Price Modeling context, FastAPI can replace or complement Flask for scenarios requiring higher throughput, async request handling, or automatic API documentation. It's particularly useful when scaling beyond 200 concurrent requests or when integrating with async data sources.

How do Server-Sent Events (SSE) enhance the user experience?

Server-Sent Events enable the server to push real-time updates to client applications over a single HTTP connection. For price predictions that may take several seconds to process, SSE allows the system to send progress updates, intermediate results, or status messages to the client. This improves user experience by providing feedback during longer-running predictions rather than forcing clients to wait for a single response.

What is the Databricks Jobs API used for?

The Databricks Jobs API allows the system to offload complex, long-running, or batch prediction workloads to Databricks compute clusters when needed. While the primary architecture avoids cluster spin-up for individual requests, some scenarios like bulk price predictions for thousands of parts or model retraining may benefit from cluster-based execution. The Jobs API provides this flexibility while maintaining the API-based interface for clients.

How does the system handle what-if scenarios?

What-if scenario analysis allows users to test different assumptions and parameters to see how they affect predicted prices. Users can modify variables like quantity, delivery timeline, supplier selection, market conditions, or part specifications through the API. The system processes each scenario independently, invoking the appropriate ML models with the modified parameters and returning predictions that help inform procurement decisions.

Horizontal scaling is achieved by deploying multiple instances of the Flask application behind a load balancer, using stateless API design so any instance can handle any request, implementing distributed caching with Redis or similar for SKU data sharing across instances, and leveraging Databricks Model Serving which automatically scales to handle increased request volume. This architecture supports growth from 200 to thousands of concurrent users without major redesign.

What monitoring and observability features are included?

How are ML models updated without downtime?

Databricks Model Serving supports blue-green deployments where new model versions are deployed alongside existing versions, traffic is gradually shifted to the new version while monitoring performance, rollback is instant if issues are detected, and old versions are retired only after the new version is validated. This ensures continuous availability while enabling regular model improvements based on new data and market conditions.

What data is required to make a price prediction?

A typical price prediction request requires part or SKU identifier, quantity or volume being procured, desired delivery timeline, supplier preferences or constraints, current market conditions or indicators, and any specific scenario parameters for what-if analysis. The system enriches this input with historical data, part specifications, and market intelligence from the SQL Warehouse before invoking ML models.

How does caching improve system performance?

Caching stores frequently accessed SKU data in memory with a Time-To-Live mechanism, reducing database query latency from hundreds of milliseconds to microseconds, decreasing load on the Databricks SQL Warehouse, enabling higher throughput for concurrent requests, and improving overall user experience with faster predictions. Cache invalidation strategies ensure data freshness while maximizing performance benefits.

Can the system handle batch predictions?

Yes, the system supports batch predictions through dedicated endpoints that accept multiple prediction requests in a single API call, process requests asynchronously using job queues, return batch results when all predictions complete, and optionally use the Databricks Jobs API for very large batches that benefit from cluster-based parallel processing. This is useful for scenarios like monthly procurement planning or supplier comparison analysis.

What happens if a model serving endpoint is unavailable?

The system implements resilience patterns including retry logic with exponential backoff for transient failures, fallback to alternative models or cached predictions when available, circuit breaker patterns to prevent cascading failures, graceful degradation with informative error messages to clients, and alerting to operations teams for investigation and remediation. This ensures high availability even when individual components experience issues.
